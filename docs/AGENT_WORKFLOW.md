# Agent Workflow

Codexは各作業を `plan / observe / implement / test / refactor / prune` の順で進める。

## 1. plan

作業前に今回の範囲を短く固定する。

必ず決める:

- 対象Phase
- 対象画面
- 追加・変更するデータ
- 子ども画面に出すもの
- 子ども画面に出さないもの
- 今回やらないこと
- 完了条件

例:

```text
Phase 1 / kids kiosk
作る: 2人並び、合計タグ、アプリ貯金、目標、買えるもの
作らない: 円換算、履歴、購入リクエスト、DB連携
```

## 2. observe

実装前に現状を読む。

確認する:

- ファイル構成
- 既存コンポーネント
- 既存データモデル
- 既存スタイル
- PLAN.mdとの差分
- 変更予定ファイルに未把握の編集がないか

新規プロジェクト初回は、空のワークスペースであることと採用スタックを確認する。

## 3. implement

planで決めた範囲だけを実装する。

守ること:

- 子ども画面は表示専用
- 残高はtransactions合計
- アプリ貯金は `max(balance - physicalTokenLimit, 0)`
- 取引は削除しない
- 取り消しは逆符号の取引で表す
- Phase外の機能を入れない

## 4. test

対象Phaseに合ったテストだけを行う。Phase 1は静的UIのため、永続化・実際の記録・実際の取り消しはテスト対象にしない。

全Phase共通の最低限:

- buildが通る
- `/kids` が表示される
- 子ども画面から台帳を変更できない
- 子ども画面に円換算と履歴が出ていない

画面幅:

- スマホ幅
- タブレット縦
- タブレット横

Phase 1で確認すること:

- `/kids`, `/parent/record`, `/parent/history`, `/parent/goal` が静的に表示される
- `/kids` は2人並びのキオスクとして読める
- `/parent/record` はスマホ幅で記録しやすい見た目になっている
- `/parent/history` は履歴と取り消し導線の見た目が分かる
- `/parent/goal` は子どもごとの目標設定UIの見た目が分かる
- ボタンを押しても実データは変わらない

Phase 2以降で確認すること:

- 親の記録で残高が変わる
- アプリ貯金が正しく変わる
- 取り消しで逆取引が追加される

## 5. refactor

テスト後に必要な範囲だけ整理する。

対象:

- 重複した残高計算
- 大きすぎるコンポーネント
- 親画面と子ども画面の責務混在
- DB化時に邪魔になるデータアクセス

避ける:

- まだ使わない抽象化
- 見た目だけの大規模作り替え
- Phase外の機能追加

## 6. prune

最後に削る観点で見直す。

削る候補:

- 子ども画面の履歴
- 子ども画面の円換算
- 購入リクエスト導線
- 目標リクエスト導線
- ランキングに見える表現
- 過剰なアニメーション
- 買えない商品の大量表示
- デバッグ表示
